# 视频入库模块设计文档

## 1. 文档目标

本文档定义推荐系统 MVP 中的**视频入库模块**。
该模块的职责不是单纯把视频“上传成功”，而是把一个已经完成切片、完成 transcript 分析、完成知识点抽取的切片视频，转换为一个**可播放、可检索、可召回、可审计**的结构化数据库对象。

本模块完成后，系统应具备以下能力：

第一，稳定保存切片视频资产及其 transcript JSON 外部对象位置。
第二，把 transcript、cue 时间轴、`coarse_unit` 映射、出现证据等结构化数据写入数据库。
第三，为未来视频召回层提供可直接消费的索引入口，而不是在召回时再去临时扫描 transcript。
第四，保证入库过程幂等、可替换、可追溯、可重试。
第五，让 AI agent 能以确定性的方式实现，不依赖模糊的人工判断或高耦合逻辑。

------

## 2. 设计背景

当前产品的推荐逻辑不是传统信息流分发，而是**学习目标驱动的内容召回**。
未来的视频召回层，其输入不是自由搜索词，而是一组应该学习或复习的 `semantic.coarse_unit`。因此，视频入库时不能只保存“视频文件”和“一份 transcript 文本”，而必须进一步把视频与知识点之间的关系结构化地提前落库。

换句话说，未来召回要回答的问题不是：

“有哪些视频里提到过这个单词？”

而是：

“对于这些当前需要学习的 coarse unit，哪些切片视频最适合作为学习承载内容？”

这要求入库阶段就要准备好如下数据：

- 一个切片视频包含哪些 coarse unit
- 每个 coarse unit 在该视频中出现了多少次
- 它是核心内容还是顺带提及
- 它出现在哪些时间位置
- 证据 cue 是哪些
- 这个视频作为该知识点学习载体的质量如何

如果这些数据不在入库阶段落好，后面的召回层就会退化成在线文本解析，性能、质量和可解释性都会很差。

------

## 3. 核心约束

## 3.1 只存切片，不存原始视频

数据库中，视频主表里的每一行都代表一个**切片视频**。
原始长视频不入库，不作为实体存在，不承担任何外键父节点角色。

这意味着：

- `video_id` 对应的是最终可播放、可推荐、可被用户消费的切片
- 不存在 `raw_video_id -> clip_video_id` 的父子关系树
- 所有召回、排序、曝光、用户行为都围绕切片 `video_id` 展开

## 3.2 `parent_video_name` 仅作来源信息

原始视频只保留一个来源字符串字段，例如原始视频标题、原文件名、上游来源名。
它的职责是：

- 来源追踪
- 多样性控制
- 去重辅助
- 排障审计

它不承担关系建模职责，不会有原始视频主表。

## 3.3 文件在存储桶，数据库只存结构化索引和路径

以下对象放入存储桶：

- 切片视频文件
- transcript JSON 文件

数据库不以大 JSON 或媒体正文为主读取路径。数据库只保存：

- 对象路径
- 哈希摘要
- 必要的结构化字段
- 召回所需的索引化结果

## 3.4 入库结果必须直接兼容未来召回层

虽然本文档只聚焦入库模块，但数据设计必须直接支持未来召回。因此本模块落库内容必须覆盖：

- 视频资产主记录
- transcript 汇总快照
- cue 时间轴
- `coarse_unit -> video_id` 的显式关系索引
- coarse unit 的证据与质量特征
- 可选的 mention 级明细
- 可追溯的入库审计

------

## 4. 设计目标与非目标

## 4.1 设计目标

本模块的核心目标是把“上游分析结果”变成“推荐系统基础索引层”。

这包括五个目标：

一是**对象清晰**。数据库里每个 `video_id` 都是一个切片视频，不引入额外的“片段子实体”造成语义混乱。
二是**读路径明确**。未来召回应能直接从 `coarse_unit_id` 找到候选 `video_id`。
三是**写路径确定**。单个切片的一次分析结果以快照方式替换写入，而不是细粒度 patch。
四是**质量可控**。不只存“包含某个词”，还要存它的角色、显著性、教学承载度和证据。
五是**工程友好**。设计必须适合用 Go + PostgreSQL/Supabase + 原生 SQL + pgx 落地。

## 4.2 非目标

本模块不负责以下内容：

- 不负责原始视频切片
- 不负责 ASR、LLM 抽取或 NLP 分析本身
- 不负责学习调度
- 不负责视频召回排序
- 不负责学习任务编排
- 不负责用户观看行为采集
- 不负责 learning engine 状态维护

本模块只负责把“一个切片视频及其分析结果”正确、完整、可追踪地放入数据库与存储系统。

------

## 5. 从未来召回倒推的核心读路径

这部分是整个设计最重要的出发点。
数据库表不是凭感觉建出来的，而是从未来最关键的查询需求倒推出来的。

### 查询一：给定一组 `coarse_unit_id`，快速找到候选切片视频

这是未来召回层的起点。
系统需要支持：

- 输入多个目标 coarse unit
- 快速找到所有命中这些 unit 的切片视频
- 按角色、显著性、教学承载度、置信度进行粗排

这要求数据库必须存在一个显式的**视频-知识点索引表**。

### 查询二：给定 `video_id + coarse_unit_id`，取该知识点在视频中的覆盖详情

未来召回层不只需要知道“命中了”，还需要知道：

- 出现了几次
- 首次出现和最后出现在哪
- 证据 cue 是哪些
- 该知识点在此视频里是不是主要内容
- 教学适配度怎样

这要求数据库不只是一个 `video_id <-> unit_id` 的布尔映射，而要存覆盖度和质量信息。

### 查询三：给定 `video_id`，取切片 transcript 快照与 cue 时间轴

未来前端和召回解释层会需要：

- 展示 transcript
- 高亮 cue
- 跳到某个时间点
- 展示“为什么推荐这个视频”

因此，入库模块必须落一份切片级 transcript 快照和 cue 级时间轴。

### 查询四：给定 `video_id + coarse_unit_id`，追踪更细粒度 mention 证据

这不是 MVP 最基础必需项，但强烈建议提前入库。
因为未来一旦要支持：

- 单词点击定位
- 片段解释
- 证据抽样
- 更细的索引质量诊断

mention 明细就会非常有价值。

## 6. 总体架构

视频入库模块可以分成四层。

### 6.1 输入层

输入来自上游内容处理流水线。
每个切片会生成一份入库 payload，包含：

- 切片视频文件对象信息
- transcript JSON 对象信息
- transcript 汇总
- cue 列表
- course unit 抽取结果
- 可选 mention 列表
- 分析任务元数据

### 6.2 存储层

视频文件与 transcript JSON 写入存储桶。
数据库只记录：

- 对象路径
- 哈希
- 元数据
- 结构化索引结果

### 6.3 索引落库层

把 transcript、cue、unit 映射、mention 等结果落入 `catalog` schema 中的多张表。
这里不再做在线分析，只做校验、幂等判断和事务写入。

### 6.4 审计与控制层

通过 job 表、状态字段、checksum、版本号和 `analysis_emitted_at` 机制，保证：

- 重复投递不出脏数据
- 旧分析不能覆盖新分析
- 任意一次失败可以追溯原因
- 入库行为可重放、可验收

------

## 7. 关键设计决策

## 7.1 为什么不采用 `video_segments` 子实体模型

有一种常见设计是：`catalog.videos` 存视频主对象，再建 `catalog.video_segments` 存片段。
这个方案在“原始视频在库、切片是其子实体”的场景下是合理的。

但你当前已经明确：

- 原始视频不入库
- 数据库里只存切片视频

在这个前提下，如果再建一个 `video_segments`，就会变成“切片视频内部再切片”，对象层级失真，后续召回和用户行为也会更复杂。

因此本设计明确选择：

**切片视频本身就是一等视频实体。**

也就是说：

- `catalog.videos.video_id` 就是未来召回对象
- transcript、cue、unit 索引作为 side tables 挂在这个 `video_id` 之下
- 不额外引入中间片段子表作为核心主模型

## 7.2 为什么不把 transcript 和 unit 结果全塞进 JSONB

因为未来读路径是明确的、结构化的：

- 按 `coarse_unit_id` 查候选视频
- 按 `video_id` 查 cue 时间轴
- 按 `video_id + coarse_unit_id` 查覆盖详情

这些场景如果只靠 JSONB，会带来：

- 索引复杂
- SQL 可维护性差
- 查询可解释性弱
- agent 更容易写错

因此本设计只把 transcript JSON 原文放存储桶，数据库中保存结构化展开结果。

## 7.3 为什么采用快照式 replace 写入

同一个切片的一次分析结果，本质上是一个完整快照。
如果做细粒度 merge，很容易残留旧 cue、旧 unit、旧 evidence，最终污染召回候选池。

因此本设计明确采用：

**单切片、单版本、单事务、replace 写入。**

即某个 `video_id` 的 transcript、cue、unit index、mention 明细在一次事务中整体替换。

## 7.4 为什么要存“质量特征”而不只是布尔映射

如果只存 `video_id <-> coarse_unit_id` 的存在关系，那么未来召回层只能回答：

“这个视频里出现过这个词。”

但真正需要的是：

“这个视频适不适合用来学这个词。”

因此，索引表必须至少包含：

- `role`
- `salience_score`
- `teachability_score`
- `confidence_score`

必要时还可以扩展语境等级字段。

## 8. 存储桶设计

## 8.1 存储对象

每个切片至少对应两个外部对象：

- 视频文件
- transcript JSON 文件

推荐使用同一个 Supabase Storage 项目中的两个逻辑前缀，或者两个 bucket。
MVP 阶段为了简化权限和管理，可以先采用一个 bucket + 两个前缀。

### 建议路径前缀

视频文件：

```
clips/{parent_video_slug}/{video_id}/video.mp4
```

transcript JSON：

```
clip-transcripts/{parent_video_slug}/{video_id}/transcript.json
```

其中：

- `parent_video_slug` 是 `parent_video_name` 规范化后的来源标识
- `video_id` 是数据库主键 UUID
- 文件名固定，便于后续规则化读取

## 8.2 存储对象不可变原则

视频文件一般是静态对象，一旦上传成功，不再覆盖。
transcript JSON 如果未来有新版本分析，推荐继续使用相同逻辑路径，但由数据库中的：

- `analysis_version`
- `transcript_sha256`
- `analysis_emitted_at`

来区分版本。

也可以将 transcript JSON 路径加版本号，但 MVP 阶段没必要把路径体系设计得过重。

## 10. 数据库模型总览

本设计建议所有视频索引相关表统一放在 `catalog` schema 下。
原因很简单：

- 它们与 `catalog.videos` 强绑定
- 它们本质上是视频内容资产的结构化索引
- MVP 阶段拆独立 schema 成本大于收益

数据库核心对象如下：

### 主表

- `catalog.videos`

### 入库审计表

- `catalog.video_ingestion_jobs`

### transcript 结构化表

- `catalog.video_transcripts`
- `catalog.video_transcript_cues`

### unit 倒排索引表

- `catalog.video_unit_index`

### mention 明细表

- `catalog.video_unit_mentions`

### 聚合视图

- `catalog.video_unit_coverage`

# 11. 数据表详细设计

## 11.1 `catalog.videos`

### 11.1.1 表职责

这张表是切片视频的**主资产表**。
数据库里每一条视频记录都对应一个最终可播放、可推荐的切片对象。

### 11.1.2 字段设计

| 字段                     | 类型                       | 说明                           |
| ------------------------ | -------------------------- | ------------------------------ |
| `video_id`               | uuid pk                    | 切片视频主键                   |
| `clip_external_id`       | text null unique           | 上游切片 ID                    |
| `parent_video_name`      | text not null              | 原始视频名                     |
| `parent_video_slug`      | text not null              | 原始视频规范化名称             |
| `clip_seq`               | integer not null           | 来源视频中的顺序               |
| `source_start_micros`    | bigint not null            | 来源起点                       |
| `source_end_micros`      | bigint not null            | 来源终点                       |
| `duration_micros`        | bigint not null            | 切片时长                       |
| `language`               | text not null default 'en' | 主语言                         |
| `title`                  | text null                  | 标题                           |
| `thumbnail_url`          | text null                  | 缩略图                         |
| `storage_video_bucket`   | text not null              | 视频 bucket                    |
| `storage_video_path`     | text not null              | 视频路径                       |
| `video_mime_type`        | text not null              | 文件类型                       |
| `video_size_bytes`       | bigint not null            | 文件大小                       |
| `video_sha256`           | text not null              | 视频哈希                       |
| `transcript_bucket`      | text not null              | transcript bucket              |
| `transcript_object_path` | text not null              | transcript 路径                |
| `transcript_sha256`      | text not null              | transcript 哈希                |
| `status`                 | text not null              | `active / inactive / failed`   |
| `media_status`           | text not null              | `uploaded / ready / failed`    |
| `analysis_status`        | text not null              | `pending / succeeded / failed` |
| `analysis_job_id`        | text null                  | 最近分析任务 ID                |
| `analysis_version`       | integer not null default 1 | 分析版本                       |
| `analysis_emitted_at`    | timestamptz null           | 分析结果时间                   |
| `content_hash`           | text not null              | 业务幂等内容哈希               |
| `created_at`             | timestamptz not null       | 创建时间                       |
| `updated_at`             | timestamptz not null       | 更新时间                       |
| `deleted_at`             | timestamptz null           | 软删除                         |

### 11.1.3 约束建议

主键：

```
primary key (video_id)
```

唯一约束建议按优先级如下：

1. `unique (clip_external_id)`，如果上游稳定提供
2. `unique (content_hash)`
3. 或者 `unique (parent_video_slug, clip_seq, source_start_micros, source_end_micros)`

### 11.1.4 索引建议

- `(parent_video_slug, clip_seq)`
- `(analysis_status, status)`
- `(language)`
- `(created_at desc)`

### 11.1.5 设计说明

这张表里不保存 transcript 正文，也不保存 unit 数组。
它只做切片视频的**资产主记录**和**生命周期状态记录**。

------

## 11.2 `catalog.video_ingestion_jobs`

### 11.2.1 表职责

记录每一次入库请求的处理过程。
它解决的是“这条数据怎么进来的、为什么成功、为什么失败、为什么被跳过”的问题。

## 12. 入库流程设计

## 12.1 总体原则

入库过程采用：

**对象先上传，数据库后索引；单切片快照，单事务替换。**

这意味着：

- 上游先把视频文件和 transcript JSON 放进 bucket
- 入库服务只接收对象路径和结构化结果
- 数据库索引写入时，以切片为单位整体 replace

## 12.2 标准流程

### 步骤 1：准备对象

上游先上传：

- 切片视频文件
- transcript JSON 文件

并生成 checksum。

### 步骤 2：发送入库请求

调用入库服务，提交 manifest。
manifest 中只包含：

- 基础元数据
- 对象路径
- transcript 汇总
- cue 列表
- unit 列表
- mention 列表
- 分析版本与时间

### 步骤 3：创建入库 job

写入 `catalog.video_ingestion_jobs`，状态置为 `running`。

### 步骤 4：幂等判断

根据现有库中同一业务对象的记录，判断这次请求是：

- 新建
- 更新
- 跳过

### 步骤 5：开启事务并 upsert `catalog.videos`

确保切片视频主记录存在且元信息最新。

### 步骤 6：replace `catalog.video_transcripts`

更新 transcript 汇总快照。

### 步骤 7：replace `catalog.video_transcript_cues`

删除旧 cue，再写新 cue。

### 步骤 8：replace `catalog.video_unit_index`

删除旧 unit 索引，再写新结果。

### 步骤 9：replace `catalog.video_unit_mentions`

删除旧 mention，再写新明细。

### 步骤 10：更新状态并结束 job

将：

- `catalog.videos.media_status = ready`
- `catalog.videos.analysis_status = succeeded`
- `catalog.videos.status = active`

并把 job 标记为 `succeeded`。

如果过程中任一步失败，则事务回滚，job 置为 `failed`。

------

## 13. 幂等与版本控制设计

这是最关键的可靠性部分。

## 13.1 业务对象幂等

用于识别“这是不是同一个切片”。

优先级建议如下：

1. `clip_external_id`
2. `content_hash`
3. `(parent_video_slug, clip_seq, source_start_micros, source_end_micros)`

如果上游有稳定 `clip_external_id`，优先用它。
否则采用内容哈希或自然业务键。

## 13.2 分析结果幂等

用于识别“这是同一个切片的新分析，还是重复投递”。

主要依据：

- `analysis_emitted_at`
- `analysis_version`
- `transcript_sha256`
- `request_payload_checksum`

## 13.3 覆盖规则

### 规则一

如果 incoming `analysis_emitted_at` 早于数据库当前值，直接跳过，不允许旧结果覆盖新结果。

### 规则二

如果 `analysis_emitted_at` 相同且 `request_payload_checksum` 相同，视为重复投递，直接跳过。

### 规则三

如果 `analysis_emitted_at` 更晚，允许 replace 写入。

### 规则四

如果时间相同但 `analysis_version` 更高，可以允许覆盖，但必须在 job 中记录原因。

------

## 14. 校验规则

入库前必须有严格校验，否则脏数据会直接污染召回基础索引。

## 14.1 基础字段校验

必须保证：

- `parent_video_name` 非空
- `clip_seq >= 0`
- `duration_micros > 0`
- `source_end_micros > source_start_micros`
- `video_object_path` 非空
- `transcript_object_path` 非空
- `video_sha256` 非空
- `transcript_sha256` 非空

## 14.2 transcript 校验

- `transcript_text` 不应为空字符串
- `cue_count` 必须等于 cue 列表长度
- 每个 cue 必须满足 `end_micros > start_micros`
- cue 时间轴应按顺序递增
- 最后一条 cue 的 `end_micros` 不应明显越过 `duration_micros`

## 14.3 coarse unit 校验

- `coarse_unit_id` 必须存在于 `semantic.coarse_unit`
- `mention_count >= 0`
- `salience_score`、`teachability_score`、`confidence_score` 均在合法范围
- `role` 必须属于允许枚举
- `context_relevance` 若存在，必须属于允许枚举
- `evidence_cue_nos` 中的 cue 号必须真实存在

## 14.4 mention 校验

- `mention_no` 在 `(video_id, coarse_unit_id)` 下唯一
- `cue_no` 必须对应现有 cue
- mention 的时间必须落在视频或所属 cue 的合法范围内

------

## 15. 状态机设计

为了便于排障和运维，主表状态要清晰。

## 15.1 `media_status`

建议枚举：

- `uploaded`
- `ready`
- `failed`

含义是：

- `uploaded` 表示对象已存在于 bucket，但业务尚未完全可用
- `ready` 表示可供业务使用
- `failed` 表示媒体侧有问题

## 15.2 `analysis_status`

建议枚举：

- `pending`
- `succeeded`
- `failed`

表示 transcript 和索引是否构建成功。

## 15.3 `status`

建议枚举：

- `active`
- `inactive`
- `failed`

其中只有 `active` 才允许未来召回层消费。

------

## 16. 性能与索引策略

MVP 阶段不追求极致复杂优化，但必须保证关键查询足够快。

### 核心原则

未来最频繁的读路径是：

- 按 `coarse_unit_id` 找视频
- 按 `video_id` 找 cue
- 按 `video_id + coarse_unit_id` 查覆盖明细

因此索引重点应放在：

- `catalog.video_unit_index`
- `catalog.video_transcript_cues`

### 不建议的做法

当前不建议把 transcript 当全文检索主入口。
全文搜索未来可以作为补充能力，但不该成为 MVP 召回主干。

## 17. 失败恢复与重试策略

## 17.1 错误分类

### 输入错误

例如：

- 缺字段
- 路径不存在
- `coarse_unit_id` 不存在
- cue 时间错乱

这类错误应直接失败，不自动无限重试。

### 暂时性错误

例如：

- bucket 超时
- 数据库瞬时连接失败
- 事务锁冲突

这类允许有限重试。

### 版本冲突

例如：

- 旧结果晚到
- 重复投递

这类应标记为 `skipped`，不视为失败。

## 17.2 半成品防护

只要 transcript、cue、unit index、mention 的写入在同一事务内进行，就不会产生“部分成功”的脏状态。

真正的边界点只有一个：

**对象上传必须先成功，再触发数据库入库。**

也就是说，本模块默认处理的是“对象已经存在”的前提，而不是边上传边建表。

## 19. 验收标准

## 19.1 功能验收

给定一个合法切片 payload，系统能：

- 成功识别视频对象存在
- 成功识别 transcript JSON 存在
- 成功写入 `catalog.videos`
- 成功写入 `catalog.video_transcripts`
- 成功写入 `catalog.video_transcript_cues`
- 成功写入 `catalog.video_unit_index`
- 成功写入 `catalog.video_unit_mentions`
- 将视频状态置为 `active + ready + succeeded`

## 19.2 幂等验收

对同一 payload 连续提交多次：

- 不产生重复视频主记录
- 不产生重复 cue
- 不产生重复 unit 索引
- 数据内容保持一致

## 19.3 版本验收

先提交新版本，再提交旧版本：

- 旧版本不得覆盖新版本
- job 必须记录为 `skipped`
- 数据库索引保持新版本不变

## 19.4 数据质量验收

随机抽样检查：

- cue 数与 transcript JSON 一致
- 所有 `coarse_unit_id` 均存在
- evidence cue 可追溯
- mention 与 unit 覆盖统计基本一致

## 19.5 可查询性验收

给定任意 `coarse_unit_id`，系统可以通过 `catalog.video_unit_index` 快速查到候选视频列表。
这是未来视频召回层能否正常工作的最基本验收条件。

------

## 20. 最终设计结论

本设计的核心选择可以概括成一句话：

**以切片视频本身作为一等视频实体，以 `catalog.videos` 为主表，以 transcript、cue、video-unit index、mentions 作为 side tables，以单切片快照式 replace 写入保证确定性，并在 `catalog` schema 内建立未来视频召回所需的结构化索引层。**

它吸收了参考设计中真正有价值的部分：

- 查询驱动建模
- 使用 `catalog` schema
- 保留 `video_unit_coverage` 聚合层
- 强调语境质量分级
- 强调单对象单事务写入

但没有采用不适合你当前前提的那部分：

- 不引入 `video_segments` 作为核心对象
- 不把原始视频建成实体
- 不让“切片之下再切片”的层级污染模型

因此，这一版设计既与当前项目约束一致，也为未来召回模块留出了清晰、稳定的接口。

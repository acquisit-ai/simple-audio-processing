# 题目生成流程

本文面向第一次接触 `9question-generation-deepseek.py` 的新人，解释这个脚本从 mapped transcript JSON 到题目 JSON 的完整语义链路。

一句话概括：

```text
它不是重新理解视频字幕，而是在已有 coarse_unit 映射结果上，筛出适合练习的上下文词项，让 AI 只生成题目内容，再由脚本补齐稳定的题目草稿字段。
```

---

## 一、脚本定位

`9question-generation-deepseek.py` 位于 `5agent-mapping-deepseek.py` 之后。

前一个脚本负责：

- 把 transcript token 合并成学习导向的 semantic token；
- 给 token 写 `explanation`、`semantic_element.translation`、`semantic_element.dictionary`；
- 尽可能把 token 映射到 `semantic.coarse_unit`；
- 在成功映射时写入 `coarse_id`、`kind`、`pos` 和稳定释义。

题目生成脚本只处理已经映射好的结果：

- 只考虑 `semantic_element.coarse_id != null` 的 token；
- 只生成视频上下文题；
- 不直接写数据库；
- 输出入库前题目草稿 JSON；
- 默认题目状态是 `draft`。

整体位置如下：

```mermaid

    raw["原始 transcript JSON"] --> mapping["5agent-mapping-deepseek.py"]
    mapping --> mapped["mapped JSON - 含 coarse_unit 映射"]
    mapped --> question["9question-generation-deepseek.py"]
    question --> output["questions JSON - pre-ingest draft"]
    output -.-> ingest["后续 ingest 脚本补 video_id"]
    ingest -.-> catalog["catalog.questions"]

```

## 二、运行入口

单文件生成：

```bash
.venv/bin/python 9question-generation-deepseek.py \
  resource/The\ Office\ BD/mapped/clip1.json \
  resource/The\ Office\ BD/questions/clip1.json
```

常用参数：

| 参数 | 默认值 | 语义 |
| --- | --- | --- |
| `mapped_json` | 无 | 输入 mapped JSON 文件。 |
| `output_questions_json` | 无 | 输出题目 JSON 文件。 |
| `--question-types` | `context_meaning_choice,context_cloze_choice` | 允许生成的题型。 |
| `--batch-size` | `10` | 每次交给 AI 的候选数量。 |
| `--selection-model` | `deepseek-v4-flash` | Candidate 筛选阶段用于最佳句子选择的模型。 |
| `--selection-top-k` | `5` | 每个 `coarse_id` 最多给 Flash 看的候选句子数量。 |
| `--selection-batch-size` | `10` | 每次 Flash 选句调用最多处理多少个 `coarse_id` group。 |
| `--selection-max-workers` | `4` | Flash 选句阶段并发调用数。 |
| `--env-path` | `.env` | 读取 `DEEPSEEK_API_KEY` 和可选 `DEEPSEEK_BASE_URL`；strict Tool Calls 默认使用 `https://api.deepseek.com/beta`。 |
| `--model` | `deepseek-v4-pro` | 题目生成模型。 |

批量生成用 `9question-generation-batch.py`。它的默认输入目录是 `resource/The Office BD/mapped`，默认输出目录是 `resource/The Office BD/questions`。

## 三、输入语义

输入必须是 `5agent-mapping-deepseek.py` 的正式输出结构。题目脚本依赖以下字段：

```json
{
  "sentences": [
    {
      "index": 14,
      "text": "The most sacred thing I do is care and provide for my workers.",
      "translation": "我做的最神圣的事情，就是关心并供养我的员工。",
      "start": 17621,
      "end": 30899,
      "tokens": [
        {
          "index": 17,
          "text": "sacred",
          "explanation": "在这里是比喻用法，表示非常重要、不可轻视。",
          "start": 26120,
          "end": 26489,
          "semantic_element": {
            "coarse_id": 138446,
            "base_form": "sacred",
            "translation": "神圣（宗教或比喻）",
            "dictionary": "表示被视为神圣或应受特殊、不可侵犯的尊重。",
            "kind": "word",
            "pos": "adjective",
            "reason": "语义可靠匹配。"
          }
        }
      ]
    }
  ]
}
```

关键含义：

- `sentence.index`：题目定位到哪一句字幕。
- `sentence.text`：上下文题展示的原句来源。
- `sentence.translation`：帮助 AI 理解上下文，不直接写入最终题目字段。
- `sentence.start/end`：最终写入 `context_start_ms/context_end_ms`。
- `token.index`：最终写入 `context_span_index`。
- `token.text`：清理后作为 `target_text`。
- `token.explanation`：帮助 AI 判断当前语境下的含义。
- `semantic_element.coarse_id`：学习系统真正追踪的 unit id。
- `semantic_element.translation`：给 AI 的简短中文标签，可作为正确选项参考。
- `semantic_element.dictionary`：给 AI 的稳定词典释义，用于理解但通常不直接展示。

这里有一个重要设计：脚本不是按原始字幕每 3 到 4 句交给 AI，而是先全局筛出候选 occurrence，再让 Flash 模型为每个 `coarse_id` 选择最适合出题的句子，最后按稳定 candidate 列表分批生成题目。candidate checkpoint 会写入 temp，保证题目生成中断后续跑时不重新选句、不改变 candidate_id。

## 五、Candidate 筛选阶段

Candidate 筛选阶段由基础硬筛、句子候选分组、Flash 选句和 candidate checkpoint 四部分组成。

基础硬筛不是“判断题目怎么写”，而是先把明显不适合交给 AI 的 token 排除掉。

### 1. 必须满足的条件

一个 token 要进入候选池，至少要满足：

- `semantic_element` 是对象；
- `semantic_element.coarse_id` 是非空整数；
- sentence 有 `index/text/translation/start/end`；
- token 有 `index/text/start/end`；
- `semantic_element.translation` 非空；
- `semantic_element.dictionary` 非空；
- `target_text` 清理后不是空字符串。

### 2. 直接拒绝的情况

以下情况会被硬筛拒绝，并写入审计：

- `coarse_id is null`
- `semantic_element is missing`
- `token timing is missing`
- `target text is empty`
- `target text is punctuation only`
- `target text is numeric`
- `semantic translation or dictionary is missing`
- `target is a low-value function word`
- `target looks like a proper name`

这些拒绝不需要 AI 判断，因为它们属于结构或低学习价值问题。

### 3. 打分与排序

进入粗候选后，脚本用 `score_candidate()` 给每个候选打分。

打分倾向：

- `kind = phrase` 权重最高；
- `adjective`、`adverb`、`verb` 优先于普通 `noun`；
- 多词 target 加分；
- 句子长度在 4 到 22 个词之间加分；
- 太短句和过长句降权。

之后按 `coarse_id` 分组，并在每个组内按句子聚合：

- 同一个 `(coarse_id, sentence_index)` 只保留一个句子候选；
- 如果同一个句子内同一个 `coarse_id` 出现多次，默认使用 `token_index` 最小的 token 代表该句；
- 每个 `coarse_id` group 最多给 Flash top K 个句子候选，默认 K 为 5；
- Flash 选句默认每 batch 处理 10 个 `coarse_id` group；
- 多个 Flash batch 会用线程池并发执行，默认并发数为 4；
- Flash 只负责从每个 group 中选择一个最佳句子，不负责 rejection。

Flash 选句使用 DeepSeek strict Tool Calls，工具名是 `submit_context_selection_batch`。输出只允许 `selections`，不允许 `rejections`。每个 group 必须选择且只能选择一个 `sentence_candidate_id`。

Flash 选句后，脚本把选中的句子候选整理成稳定 `QuestionCandidate` 列表，并写入 temp 的 `candidate_checkpoint`。后续题目生成阶段只读取这个 checkpoint，避免中断续跑时重新选句导致 candidate_id 变化。

这里的粒度是唯一 `coarse_id`，不是每一次出现的 occurrence。正常由当前 mapping 脚本生成的 mapped JSON 中，所有实际出现的 `coarse_id != null` 会在筛选阶段按 `coarse_id` 聚合，并在最终输出的 `selected_coarse_unit_refs.refs` 中形成严格一对一关系：

```text
一个唯一 coarse_id -> 一个 selected_coarse_unit_ref
```

如果同一个 `coarse_id` 在全文出现多次，Flash 只从这些 occurrence 对应的候选句子里选一个最适合出题的句子作为 ref。

候选生命周期如下：


## 六、AI 输入与输出边界

这是脚本最核心的职责边界。

AI 只负责生成题目内容。脚本负责所有稳定元数据。

### 1. AI 能看到什么

每次 AI batch 的 user message 由两部分组成：

1. `FULL_VIDEO_TRANSCRIPT`：整段视频的英文全文，每个 batch 完全一致，用作 DeepSeek prefix cache 友好的背景上下文；
2. `CURRENT_CANDIDATE_BATCH`：当前 batch 的候选 JSON。

`QuestionCandidate.to_ai_payload()` 只给 AI 这些 candidate 字段：

```json
{
  "candidate_id": "c_000001",
  "target_text": "sacred",
  "base_form": "sacred",
  "coarse_label": "神圣（宗教或比喻）",
  "kind": "word",
  "pos": "adjective",
  "sentence_text": "The most sacred thing I do is care and provide for my workers.",
  "token_explanation": "在这里是比喻用法，表示非常重要、不可轻视。"
}
```

AI 看不到：

- `sentence_translation`
- `coarse_definition`
- `coarse_unit_id`
- `sentence_index`
- `token_index`
- `start/end`
- `status`
- 原始 mapped JSON 的其他 token

这样做是为了避免 AI 篡改稳定字段，也减少无关上下文干扰。

### 2. AI 必须提交什么

AI 不直接返回正文 JSON，而是必须调用 `submit_question_batch` 工具。脚本使用 DeepSeek strict Tool Calls，`QUESTION_OUTPUT_TOOL.function.strict = true`，默认 base URL 是 `https://api.deepseek.com/beta`。

字段结构由代码里的 `QUESTION_OUTPUT_TOOL` 定义，prompt 不再重复完整 schema；工具的 `arguments` 由 `AIQuestionBatchOutput` 校验：

```json
{
  "results": [
    {
      "candidate_id": "c_000001",
      "question_type": "context_meaning_choice",
      "content_payload": {
        "question": "这里的 “sacred” 最接近什么意思？",
        "context_text": "The most sacred thing I do is care and provide for my workers.",
        "options": [
          { "id": "correct", "text": "神圣、非常重要" },
          { "id": "wrong_1", "text": "普通、随便" },
          { "id": "wrong_2", "text": "昂贵、奢侈" },
          { "id": "wrong_3", "text": "快速、临时" }
        ],
        "explanation": "sacred 在这里是比喻用法，表示说话人认为这件事非常重要、不可轻视。"
      }
    }
  ],
  "rejections": [
    {
      "candidate_id": "c_000002",
      "reason": "上下文不足，不适合生成高质量题目。"
    }
  ]
}
```

AI tool arguments 禁止额外字段。比如它如果输出 `coarse_id`、`context_start_ms`，Pydantic 会因为 `extra="forbid"` 直接拒绝。

输出语言要求：

- `question` 必须是中文；
- `context_meaning_choice` 的 `options[].text` 必须是中文释义；
- `context_cloze_choice` 的 `options[].text` 应使用英文单词或短语，因为它们用于补全英文上下文；
- `explanation` 必须是中文，且只解释正确选项为什么正确；
- `context_text` 保持英文视频原句或英文 cloze 上下文。

### 3. 交互时序

```mermaid
sequenceDiagram

```

## 七、脚本补齐的最终题目字段

`merge_ai_result()` 是 AI 结果变成最终题目的关键函数。

AI 只给 `content_payload`，脚本从 `QuestionCandidate` 补齐稳定草稿字段：

| 最终字段 | 来源 |
| --- | --- |
| `scope_type` | 固定写 `video_unit`。 |
| `question_type` | AI 输出，但必须在本次允许题型中。 |
| `coarse_unit_id` | 候选的 `coarse_unit_id`。 |
| `target_text` | 候选的清理后 token 文本。 |
| `context_sentence_index` | 候选的 `sentence_index`。 |
| `context_span_index` | 候选的 `token_index`。 |
| `context_start_ms` | 候选句子的 `start`。 |
| `context_end_ms` | 候选句子的 `end`。 |
| `content_payload` | AI 输出并经脚本校验。 |
| `status` | 固定写 `draft`。 |

## 八、内容校验规则

`validate_content_payload()` 对 AI 生成的题目内容做二次校验。

当前规则：

- `question` 必须非空；
- `options` 必须正好 4 个；
- option id 必须严格等于：

```json
["correct", "wrong_1", "wrong_2", "wrong_3"]
```

- 每个 option 的 `text` 必须非空；
- 4 个 option 文本忽略大小写后不能重复；
- `context_text` 必须非空；
- `context_cloze_choice` 的 `context_text` 必须包含 `____`；
- `context_cloze_choice` 的 `context_text` 不能泄露 `target_text`。

这里的语义是：AI 可以写题，但题目能不能进入最终 JSON，由脚本决定。

```mermaid
    ids -->|是| type{"question_type 允许？"}
    type -->|否| reject["validation_reject"]
    type -->|是| payload{"content_payload 合法？"}
    payload -->|否| reject
    payload -->|是| merge["merge_ai_result"]
    merge --> final["FinalQuestion"]

    classDef decision fill:#fff4e6,stroke:#e67700,color:#5f370e;
```

## 九、最终输出文件

最终输出不是裸数组，而是一个带来源和审计摘要的对象。

```json
{
  "source": {
    "mapped_json": "resource/The Office BD/mapped/clip1.json",
    "model": "deepseek-v4-pro"
  },
  "questions": [],
  "audit": {
    "candidate_count": 46,
    "generated_count": 28,
    "rejected_count": 18
  },
  "selected_coarse_unit_refs": {
    "version": 1,
    "selection_model": "deepseek-v4-flash",
    "selection_top_k": 5,
    "allowed_question_types": [
      "context_meaning_choice",
      "context_cloze_choice"
    ],
    "refs": [
      {
        "coarse_unit_id": 130328,
        "sentence_index": 0,
        "token_index": 2
      }
    ]
  }
}
```

字段含义：

- `source.mapped_json`：本次生成使用的 mapped JSON 路径。
- `source.model`：题目生成模型。
- `questions`：入库前题目草稿；后续 ingest 阶段再补数据库 `video_id`。
- `audit.candidate_count`：进入 AI 候选池的候选数。
- `audit.generated_count`：最终成功生成的题目数。
- `audit.rejected_count`：AI 拒绝数加脚本 validation 拒绝数。
- `selected_coarse_unit_refs`：第一阶段筛选和 Flash 选句后的轻量引用列表，放在顶层末尾。每个 ref 只保存 `coarse_unit_id`、`sentence_index`、`token_index`，完整候选信息可回到 mapped JSON 查。正常 mapping 输出中，全文每个唯一 `coarse_id != null` 在这个 `refs` 列表里严格对应一条 ref；同一个 `coarse_id` 多次出现时也只保留一条由 Flash 选出的最佳上下文 ref。

`selected_coarse_unit_refs.refs` 的核心语义是：在正常由当前 mapping 脚本生成、字段完整的 mapped JSON 中，文中出现的每一个唯一 `coarse_id` 都会在 ref 列表中严格一对一出现。一个 `coarse_id` 可以在全文有多个 occurrence，但第一阶段会把它们聚合成同一个 group，并最终只选择一个最适合出题的句子作为该 `coarse_id` 的代表 ref。

注意：`hard_filter_reject` 不计入最终 wrapper 的 `rejected_count`，它们写入详细审计日志。

## 十、审计日志

脚本会在输出目录下创建：

```text
log/<mapped_json_name>.question_audit.jsonl
```

审计事件类型：

| event | 什么时候写入 |
| --- | --- |
| `hard_filter_reject` | 代码硬筛阶段拒绝 token。 |
| `resume_loaded` | 启动时读取到正式输出或 temp 续跑状态。 |
| `ai_rejection` | AI 主动拒绝某个候选。 |
| `validation_reject` | AI 给了结果，但脚本校验不通过。 |
| `accepted_question` | 题目成功进入最终输出。 |

示例：

```json
{"event":"accepted_question","candidate_id":"c_000001","question_type":"context_cloze_choice","coarse_unit_id":130328,"target_text":"provide for"}
```

审计日志的价值是排查“为什么某个 token 没出题”：

- 如果没有进入候选池，看 `hard_filter_reject`；
- 如果 AI 认为不适合，看 `ai_rejection`；
- 如果 AI 生成内容有问题，看 `validation_reject`；
- 如果成功生成，看 `accepted_question`。

## 十一、中间文件与原子写入

脚本和 mapping 流程一样，采用中间文件加原子替换。

目录结构：

```text
output_dir/
  questions.json
  temp/
    questions.json
  log/
    mapped.json.question_audit.jsonl
```

每个 AI batch 结束后：

1. 首次运行 Candidate 筛选阶段后，先写入 `candidate_checkpoint`，包含 Flash 选句后的稳定 `QuestionCandidate` 列表；
2. 题目生成每个 batch 结束后，构造当前累计 `FinalOutput`；
3. 在 temp 输出的 `audit.processed_candidate_ids` 写入已处理 candidate，用于中断后续跑；
4. temp 输出继续保留 `candidate_checkpoint`，保证续跑不重新调用 Flash 选句；
5. 写入 `output_dir/temp/<output_name>`；
6. 下一次启动时优先读正式输出；如果正式输出不存在，就读 temp 中间文件；
7. 续跑时跳过已经接受成题的 candidate，以及 temp 里标记为 processed 的 candidate；
8. 全部 batch 完成后写入正式 `output_questions_json`；
9. 删除 temp 中间文件。

正式输出保持入库前草稿结构，同时在顶层末尾保留 `selected_coarse_unit_refs`，方便回看第一阶段选中的 coarse unit 上下文。完整 `candidate_checkpoint` 和 `processed_candidate_ids` 只存在于 temp 中，不写入正式输出。

这样做的目的是避免运行中断时留下半截正式输出。

## 十三、常见误解

### 误解一：AI 负责生成完整入库行

不是。AI 只生成 `content_payload`。稳定草稿字段由脚本补齐，数据库 `video_id` 由后续入库阶段补齐。

### 误解二：所有有 coarse_id 的 token 都会出题

不是。`coarse_id != null` 只是入门条件。脚本还会做低价值词过滤、结构校验、按 `coarse_id` 分组、Flash 选最佳句子，并保证同一个 `coarse_id` 在当前视频里最多进入 Pro 题目生成一次。

### 误解三：题目生成有最终题数上限

没有。当前版本已经删除 `--max-questions` 和内部 `candidate_pool_limit`。第一阶段会按全文所有通过基础硬筛的唯一 `coarse_id` 建立候选，第二阶段会按 `--batch-size` 分批处理所有剩余候选。最终题目数量由 AI rejection 和脚本 validation 结果自然决定。

### 误解四：`context_start_ms/context_end_ms` 来自 token 时间

不是。当前脚本使用句子 `start/end`，因为 `content_payload.context_text` 是句子级上下文。token 时间目前只参与候选结构校验和内部定位。

### 误解五：batch 是按 3 到 4 句切分

不是。mapping 脚本适合按句子批处理；题目生成脚本先全局筛候选，再按候选数量分批给 AI。

## 十四、当前边界

当前版本只做：

- `scope_type = video_unit`
- `question_type = context_meaning_choice`
- `question_type = context_cloze_choice`
- 输出 `draft` 题目 JSON
- 不直接写数据库
- 不生成通用 unit 题
- 不从 DB 查询 `video_id`
- 不从 DB 查询 distractor 池

这些边界让脚本保持清晰：它是“题目内容资产生成器”，不是题目推荐系统，也不是题库入库器。
